## Introduction

Here's a truth that many QA courses don't tell you: **you don't need to be a developer to understand the tech stack you're testing**. But you do need a working map of it.

Understanding your tech stack doesn't mean reading source code or knowing how to configure a build pipeline. It means knowing enough to answer questions like:

- "Why does this bug only appear in Safari?"
- "Could this form validation be happening in the browser or on the server?"
- "Why does the app behave differently on Android 11 vs Android 14?"
- "Which automation tool makes sense for testing this app?"

This knowledge shapes your entire testing strategy. A QA engineer who understands their stack tests smarter, communicates more effectively with developers, and picks the right tools for the job. Let's build that understanding.

## The Four Application Types and What They Mean for QA

### Web Applications

Web apps run in a browser and are the most common environment for QA engineers. But "web app" covers a huge range of architectures, and the differences matter.

**Server-Side Rendered (SSR) Apps** (traditional web, Rails, Django, Laravel):
- HTML is generated on the server for each request
- Each action often causes a full page reload
- Testing is more predictable — what you see is what the server sent
- Good for: straightforward end-to-end testing with Selenium, Playwright, Cypress
- Watch for: session handling, CSRF tokens, server-side validation

**Single-Page Applications (SPAs)** (React, Vue, Angular):
- One HTML page loads once; JavaScript handles everything after
- Navigation doesn't trigger page reloads
- State lives in the browser — can be complex to reason about
- Content may load asynchronously after the page appears "ready"
- Good for: Playwright, Cypress (both handle async well)
- Watch for: race conditions, loading states, client-side validation vs server-side validation

**Server-Side Rendering with Hydration** (Next.js, Nuxt, SvelteKit):
- First load is server-rendered; subsequent interactions are SPA-style
- Harder to reason about where validation happens
- Watch for: hydration errors, SEO-critical content, performance on first paint

**What to ask your team:**
```
"Is this a React/Vue/Angular app, or is the HTML generated by the server?
Does navigating between pages cause a full page reload, or does the URL 
change without reloading?"
```

### Mobile Applications

Mobile testing introduces constraints that don't exist on web: hardware variation, OS versions, network conditions, battery state, and app store distribution.

**Native Apps** (Swift/Objective-C for iOS, Kotlin/Java for Android):
- Built specifically for one platform
- Full access to device APIs (camera, GPS, contacts, push notifications)
- Must be tested on real devices AND simulators
- Distribution is controlled by Apple/Google review process
- Tools: XCUITest (iOS), Espresso (Android), Appium (cross-platform)

**Cross-Platform Apps** (React Native, Flutter, Xamarin):
- One codebase compiles to native or near-native on both platforms
- Behavior can differ between iOS and Android despite shared code
- Platform-specific bugs are common and frustrating
- Tools: Appium, Detox (React Native), Flutter integration tests

**Hybrid/WebView Apps** (Ionic, Cordova):
- Essentially a web app wrapped in a native container
- The "app" is a browser window
- Easier to test with web tools, but native container adds complexity
- Tools: Appium, Selenium (for the WebView), native tools for the container

**Key questions for mobile QA:**
```
"Which platforms do we support? What's our minimum OS version requirement?
Do we use any device-specific features like camera, biometrics, or GPS?
Do we have a device farm or do we test on physical devices?"
```

### API and Backend Services

Even if you're a "manual" tester, understanding the API layer is increasingly important. Modern applications are APIs with thin frontends on top. Testing the API independently means:
- Faster feedback than UI testing
- More reliable results (no browser rendering issues)
- Better coverage of edge cases and error paths
- Easier to test in isolation without a finished UI

**REST APIs:**
- Stateless, uses standard HTTP verbs (GET, POST, PUT, DELETE, PATCH)
- Data typically exchanged as JSON
- Auth usually via tokens (JWT, OAuth, API keys)
- Tools: Postman, Insomnia, Hoppscotch, curl

**GraphQL APIs:**
- Single endpoint, clients request exactly the data they need
- Queries (read) and Mutations (write)
- Can be harder to understand without schema documentation
- Tools: GraphiQL, Apollo Studio, Postman (supports GraphQL)

**gRPC / Protobuf:**
- Binary protocol, much faster than REST
- Common in microservices
- Requires more setup to test manually
- Tools: grpcurl, BloomRPC, Postman (supports gRPC)

**Key questions for API QA:**
```
"Do you have a Swagger/OpenAPI spec or a Postman collection I can use?
Where does authentication happen — in a gateway or in individual services?
Are there rate limits I should be aware of when running tests?"
```

### Desktop Applications

Desktop apps are less common in modern software but important in enterprise software, developer tools, and creative applications.

**Electron Apps** (VS Code, Slack, Discord, Figma):
- Built with web technologies (HTML/CSS/JS) in a desktop wrapper
- Can be tested with Spectron, Playwright (Electron mode)
- Updates are pushed by the vendor — must test after each update
- Watch for: file system access, system notifications, multiple window management

**Native Desktop Apps** (C++, C#/.NET, Java Swing):
- Platform-specific: Windows, macOS, Linux may all behave differently
- Automated testing is harder — tools like WinAppDriver, Accessibility Inspector
- Often have long test cycles — automation ROI is high here

## Reading Your Tech Stack Without Being a Developer

You don't have to read code to understand a tech stack. Here are practical techniques:

### Technique 1: The Browser DevTools Approach

Open Chrome DevTools (F12) while using the application:

- **Network tab**: Shows every API call being made. You can see the URL, method, request/response body, and status code. This tells you whether validation is happening in the browser or on the server.
- **Console tab**: Shows JavaScript errors. If something breaks silently, this is where you'll find the error.
- **Application tab**: Shows cookies, local storage, session storage — where the app stores your session data.

**How to use this with Zen:**
```
Prompt: I opened DevTools while testing and found this network call happening
when I click "Submit Order":

POST /api/v2/orders
Request body: {"cart_id": "abc123", "payment_token": "tok_xyz", "address_id": "55"}
Response: 422 {"error": "insufficient_inventory", "item_id": "77"}

What does this tell me about how I should test this feature?
What edge cases does this reveal that I should test?
```

### Technique 2: The Package File Approach

Most projects have a dependency file that lists their tech stack clearly. Look for:

- `package.json` — Node.js/JavaScript project, lists all dependencies
- `requirements.txt` or `pyproject.toml` — Python project
- `Gemfile` — Ruby project
- `pom.xml` or `build.gradle` — Java project
- `pubspec.yaml` — Flutter/Dart project

You don't need to understand every dependency. Share the file with Zen:

```
Prompt: Here is our package.json file. I'm a QA engineer, not a developer.
Can you tell me:
1. What is the main framework this app is built with?
2. What is used for state management?
3. What testing tools are already installed?
4. Are there any dependencies that I should be aware of from a testing perspective 
   (e.g., payment libraries, analytics, feature flags)?

[paste package.json]
```

### Technique 3: The Architecture Conversation

Schedule a 30-minute "architecture orientation" with a developer or tech lead. This is not a deep dive — it's a map reading session.

**Questions to bring:**

1. Can you draw me a simple diagram of the main components? (Frontend → Backend → Database → External services)
2. Where does data validation happen — frontend, backend, or both?
3. How does authentication work? Where are sessions/tokens stored?
4. What external services do we depend on? What happens when they go down?
5. What are the parts of the codebase that developers are most worried about?

Take notes and share them with Zen to help structure your testing approach.

## How Your Stack Affects CI/CD Tool Choices

Understanding your stack helps you advocate for the right testing tools. Here's a practical reference:

| Stack | Recommended UI Test Tool | Recommended API Test Tool | Recommended CI Platform |
|-------|--------------------------|--------------------------|-------------------------|
| React/Vue/Angular (SPA) | Playwright, Cypress | Postman, REST-assured | GitHub Actions, CircleCI |
| Next.js / SSR | Playwright, Cypress | Postman | GitHub Actions, Vercel CI |
| iOS (native) | XCUITest, Appium | XCTest, Postman | Xcode Cloud, Bitrise, Fastlane |
| Android (native) | Espresso, Appium | OkHttp tests | Firebase Test Lab, Bitrise |
| React Native | Detox, Appium | Postman | Bitrise, GitHub Actions |
| Flutter | Flutter integration tests | Dart test | GitHub Actions, Codemagic |
| REST API (any) | N/A | Postman, k6, Karate | GitHub Actions, Jenkins |
| GraphQL | Playwright (via UI) | Apollo Studio, Postman | GitHub Actions |
| Electron | Playwright (Electron) | N/A | GitHub Actions |
| .NET Desktop | WinAppDriver | N/A | Azure DevOps |

**How to use this table:**
- Find your primary application type
- Note the recommended tools
- When you're ready for the automation phase of this series, you'll have a validated starting point

## Common Stack Misconceptions for QA Engineers

### "If it passes in Chrome, it passes everywhere"

**Reality:** Browsers implement web standards differently, especially for:
- CSS layout and rendering
- JavaScript behavior at the edges (date handling, number precision)
- Web APIs (camera, microphone, geolocation)
- PDF rendering
- Form auto-fill behavior

**Minimum browser matrix for most web apps:**
- Chrome (latest, latest-1)
- Firefox (latest)
- Safari (latest) — especially if you have macOS/iOS users
- Edge (latest)

### "Mobile apps are just the website on a phone"

**Reality:** Native mobile apps are completely separate codebases from the web app. A bug in the web checkout doesn't automatically exist in the iOS app, and vice versa. They can also behave very differently because of:
- Different code (Swift vs JavaScript)
- Different network handling
- Different input methods (touch vs keyboard)
- OS-specific behaviors

### "The API is always right — it's the UI that's broken"

**Reality:** API bugs are at least as common as UI bugs, and often more damaging. A UI bug affects how users interact with the system. An API bug often corrupts data, bypasses security, or breaks integrations. Always test both layers.

## Asking Zen to Explain Technical Concepts

You will encounter technical terms you don't recognize. This is normal and expected. Zen is excellent at translating developer vocabulary into QA-relevant language.

**Prompt templates for technical explanations:**

```
Prompt: Our developers are saying the new feature uses "WebSockets." 
I'm a QA engineer, not a developer. Can you explain:
1. What WebSockets are in plain English
2. How they're different from regular HTTP requests
3. What unique testing challenges WebSockets create
4. What I should specifically test differently because of this
```

```
Prompt: Our app uses Redis for caching. What does that mean for my testing?
Specifically: if I test a feature and the cache has stale data, 
how might that affect my results? How do I account for this in my test design?
```

```
Prompt: The team mentioned we're using "feature flags" to roll out the new 
dashboard. What does this mean? How should I test features that are behind 
a feature flag? What environments and user accounts will I need?
```
